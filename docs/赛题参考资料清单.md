# AI 数字人外呼多轮对话评测系统参考资料清单

本文档整理适合本赛题借鉴的论文、Benchmark、开源项目和工程框架，重点围绕：

- 多轮对话指令遵循评测
- 用户模拟器生成外呼场景数据
- LLM-as-a-judge / Rubric Judge 自动评分
- 可解释、可量化的评估报告生成

## 1. 赛题理解对应的技术关键词

建议后续检索时优先使用以下关键词：

- Multi-turn dialogue evaluation
- Multi-turn instruction following
- Task-oriented dialogue evaluation
- User simulator for task-oriented dialogue
- LLM-as-a-judge for conversation evaluation
- Conversational G-Eval
- Rubric-based LLM evaluation
- Customer service dialogue evaluation
- Conversation trajectory evaluation
- Dialogue state tracking evaluation

## 2. 多轮指令遵循与对话评测论文

### 2.1 MultiChallenge: A Realistic Multi-Turn Conversation Evaluation Benchmark

链接：https://arxiv.org/abs/2501.17399

可借鉴点：

- 面向真实多轮对话，而不是单轮问答。
- 强调多轮中的指令保持、上下文记忆、自洽性和复杂任务完成。
- 使用 instance-level rubrics，即每个测试样例都有独立评测标准。
- 适合借鉴到本赛题中的“任务指令自动拆解为多个评分点”。

适合本赛题的用法：

- 将每条外呼任务拆成一组 rubric。
- 每个 rubric 绑定证据、得分、扣分原因。
- 不只评最终结果，还评整个对话过程。

### 2.2 MT-Eval: A Multi-Turn Capabilities Evaluation Benchmark for Large Language Models

链接：https://arxiv.org/abs/2401.16745

可借鉴点：

- 专门评估大模型多轮对话能力。
- 关注多轮场景下模型性能退化。
- 可用于设计“上下文保持、目标推进、前后一致性”等指标。

适合本赛题的用法：

- 在评测报告中加入多轮维度分析。
- 识别数字人在第几轮开始偏离任务、重复提问或遗忘用户信息。

### 2.3 Multi-IF: Benchmarking LLMs on Multi-Turn and Multilingual Instructions Following

链接：https://huggingface.co/papers/2410.15553

可借鉴点：

- 将指令遵循评测从单轮扩展到多轮。
- 强调多轮中连续指令、约束条件和语言环境的变化。
- 对中文外呼任务有参考价值。

适合本赛题的用法：

- 将业务指令拆成“必须做、禁止做、条件触发、格式要求”等可验证约束。
- 评估数字人是否在多轮中持续遵守这些约束。

### 2.4 RankJudge: A Multi-Turn LLM-as-a-Judge Synthetic Benchmark Generator

链接：https://arxiv.org/abs/2605.21748

可借鉴点：

- 针对多轮对话中的 LLM Judge 可靠性。
- 通过向对话中注入单点缺陷，构造可控评测样本。
- 用于判断评测器是否真的能识别错误。

适合本赛题的用法：

- 构造带已知问题的外呼对话，例如：
  - 漏掉身份核验
  - 错误承诺优惠
  - 未处理用户拒绝
  - 未记录用户意向
  - 未遵守结束确认流程
- 用这些样本校准自动评测模型。

### 2.5 Can Small Language Models Handle Context-Summarized Multi-Turn Customer-Service QA?

链接：https://arxiv.org/abs/2602.00665

可借鉴点：

- 直接涉及 customer-service、多轮对话、上下文摘要和 LLM-as-a-judge。
- 提到按客服阶段进行行为分析。
- 对外呼数字人的阶段式评测很有参考意义。

适合本赛题的用法：

- 将外呼流程拆成阶段：
  - 开场与身份确认
  - 任务说明
  - 用户异议处理
  - 业务确认
  - 结果记录
  - 礼貌结束
- 每个阶段分别评分。

### 2.6 PlatoLM: Teaching LLMs in Multi-Round Dialogue via a User Simulator

链接：https://arxiv.org/abs/2308.11534

可借鉴点：

- 使用用户模拟器生成多轮交互。
- 关注如何模拟更接近真实人的提问和反应。

适合本赛题的用法：

- 设计不同用户画像和行为策略。
- 让用户模拟器主动覆盖任务中的关键分支。

### 2.7 Evaluating LLM-based Agents for Multi-Turn Conversations: A Survey

链接：https://arxiv.org/abs/2503.22458

可借鉴点：

- 多轮智能体评测综述。
- 可以帮助梳理评测指标、评测对象和自动化方法。

适合本赛题的用法：

- 用于写方案背景、相关工作和技术路线综述。

## 3. 用户模拟器相关开源项目

### 3.1 OpenEvals

链接：https://github.com/langchain-ai/openevals

文档：https://github.com/langchain-ai/openevals

可借鉴点：

- 支持多轮模拟：`run_multiturn_simulation`。
- 支持 LLM 模拟用户：`create_llm_simulated_user`。
- 支持对 conversation trajectory 做 LLM-as-judge。

适合本赛题的用法：

- 参考它的“模拟用户和被测系统轮流对话”的工程结构。
- 用 task instruction + persona + scenario 生成外呼模拟对话。
- 评测完整对话轨迹，而不是单条回复。

### 3.2 prompt-based-user-simulator

链接：https://github.com/telepathylabsai/prompt-based-user-simulator

论文：https://arxiv.org/abs/2306.00774

可借鉴点：

- 基于 ConvLab-2 实现 prompt-based 用户模拟器。
- 支持 task-oriented dialogue。
- 通过 in-context learning 生成用户回复。

适合本赛题的用法：

- 借鉴用户模拟器 prompt 设计。
- 将用户状态、目标、约束、情绪写入 persona。
- 按轮生成用户反应。

### 3.3 UserSimCRS

链接：https://github.com/iai-group/UserSimCRS

可借鉴点：

- 一个可扩展用户模拟工具。
- 同时支持 agenda-based simulation 和 LLM-based simulation。
- 包含用户建模和 LLM-as-a-judge 评估。

适合本赛题的用法：

- 用户模拟器可以采用双模式：
  - Agenda-based：确保覆盖业务节点。
  - LLM-based：保证表达自然、多样。

### 3.4 Synthetic Conversation Generation

链接：https://github.com/Channel-Labs/synthetic-conversation-generation

可借鉴点：

- 生成多轮合成对话用于测试和优化。
- 强调用户 persona、多样化交互和规模化生成。

适合本赛题的用法：

- 用于设计模拟对话数据生成模块。
- 按不同用户画像批量生成外呼测试集。

### 3.5 user-satisfaction-simulation

链接：https://github.com/sunnweiwei/user-satisfaction-simulation

论文：Simulating User Satisfaction for the Evaluation of Task-oriented Dialogue Systems, SIGIR 2021

可借鉴点：

- 数据包含 6800 条任务型对话。
- 标注了 5 级用户满意度。
- 包含用户满意度解释文本。
- 覆盖 JDDC、SGD、MultiWOZ、ReDial、CCPE 等数据源。

适合本赛题的用法：

- 借鉴用户满意度评分方式。
- 在报告中增加“用户体验/用户情绪”维度。

## 4. LLM-as-Judge 与多轮评测框架

### 4.1 DeepEval Multi-Turn Evaluation

多轮评测文档：https://deepeval.com/guides/guides-multi-turn-evaluation

多轮指标文档：https://deepeval.com/guides/guides-multi-turn-evaluation-metrics

Conversational G-Eval：https://deepeval.com/docs/metrics-conversational-g-eval

可借鉴点：

- 使用 `ConversationalTestCase` 表示完整多轮对话。
- 内置多轮指标：
  - Conversation Completeness
  - Knowledge Retention
  - Role Adherence
  - Turn Relevancy
  - Conversation Relevancy
- 支持 `ConversationalGEval` 自定义自然语言评测标准。
- 支持 `ConversationalDAGMetric`，可用决策树做结构化评测。

适合本赛题的用法：

- 对业务强约束部分使用 DAGMetric 思想。
- 对话术自然度、异议处理、情绪安抚使用 G-Eval 思想。
- 输出每个指标的得分、原因和证据。

### 4.2 MLflow Multi-Turn Evaluation

链接：https://mlflow.org/docs/latest/genai/eval-monitor/running-evaluation/multi-turn/

内置 Judge 文档：https://mlflow.org/docs/latest/genai/eval-monitor/scorers/llm-judge/predefined/

可借鉴点：

- 支持多轮 conversation evaluation。
- 内置 judges：
  - ConversationCompleteness
  - UserFrustration
  - KnowledgeRetention
  - RelevanceToQuery
  - Safety
- 支持 session/thread 级评测和监控。

适合本赛题的用法：

- 将一通外呼视为一个 session。
- 对 session 级别输出总分和维度分。
- 保留 trace，方便报告追溯。

### 4.3 LangSmith Multi-Turn Evals

链接：https://docs.langchain.com/langsmith/online-evaluations-multi-turn

博客：https://www.langchain.com/blog/insights-agent-multiturn-evals-langsmith

可借鉴点：

- 将多轮对话作为 thread 进行评估。
- 强调 trajectory，即对话如何一步步展开。
- 支持线上监控与评测结合。

适合本赛题的用法：

- 报告中展示每条模拟对话的轨迹。
- 将失败归因到具体轮次和状态转移。

### 4.4 LLM-as-a-Judge 背景资料

链接：https://en.wikipedia.org/wiki/LLM-as-a-Judge

相关经典方向：

- G-Eval
- MT-Bench
- Chatbot Arena
- AlpacaEval
- Arena-Hard-Auto

可借鉴点：

- 使用强模型作为自动评委。
- 对开放式对话任务进行可扩展评测。
- 需要注意 judge 偏差、长度偏差、模型偏好和一致性问题。

适合本赛题的用法：

- 不建议只用一个 LLM 给总分。
- 更稳妥的是：
  - 规则检查
  - Rubric Judge
  - 多 judge 或多次采样一致性
  - 每个扣分点必须引用证据

## 5. 任务型对话与业务流程评测项目

### 5.1 ToolTalk

链接：https://github.com/microsoft/ToolTalk

可借鉴点：

- 面向对话式工具调用评测。
- 每轮评估模型是否预测正确的工具序列。
- 一个 conversation 包含多个子任务。

适合本赛题的用法：

- 如果数字人需要查库、记录结果、更新标签，可以评估：
  - 是否在正确轮次调用正确业务动作
  - 参数是否正确
  - 状态是否被正确更新

### 5.2 CL-Bench

链接：https://github.com/Arc-Computer/CL-Bench

可借鉴点：

- 面向 CRM 工作流、多轮对话、状态变更和跨实体关系。
- 数据包含 expected responses、initial CRM state、state mutations。
- 指标包含 conversation-level 和 turn-level success。

适合本赛题的用法：

- 外呼任务也可以建模为状态机：
  - 用户身份是否确认
  - 用户是否理解业务
  - 用户是否同意
  - 拒绝原因是否记录
  - 是否完成结束流程
- 评分不仅看文本，还看状态是否正确推进。

### 5.3 customer-service-GPT

链接：https://github.com/znat/customer-service-GPT

可借鉴点：

- LLM-first process-driven chatbot。
- 将业务逻辑嵌入 prompt，引导模型完成流程化客服任务。

适合本赛题的用法：

- 参考其流程驱动设计。
- 反向用于评测：检查数字人是否遵循了预期业务流程。

### 5.4 SimpleTOD

链接：https://github.com/salesforce/simpletod

可借鉴点：

- 经典任务型对话模型。
- 统一处理 NLU、DST、DM、NLG。
- MultiWOZ 评测中包含对话状态追踪和端到端任务完成。

适合本赛题的用法：

- 借鉴 DST，即 dialogue state tracking。
- 对外呼对话实时抽取状态：
  - 是否确认本人
  - 用户意向
  - 用户异议
  - 已说明权益
  - 是否完成记录

## 6. 可直接迁移到本赛题的系统设计

### 6.1 任务指令解析模块

输入原始任务指令，输出结构化评测标准：

```json
{
  "goal": "完成用户权益确认并记录用户意向",
  "required_steps": [
    "开场说明身份和来电目的",
    "完成用户身份核验",
    "说明权益内容",
    "确认用户是否接受",
    "记录用户意向"
  ],
  "conditional_steps": [
    {
      "condition": "用户质疑真实性",
      "expected_behavior": "解释官方来源，不施压，不承诺未授权内容"
    },
    {
      "condition": "用户拒绝",
      "expected_behavior": "礼貌确认原因并记录，不反复骚扰"
    }
  ],
  "forbidden_behaviors": [
    "承诺百分百到账",
    "诱导用户提供敏感信息",
    "在用户明确拒绝后持续施压"
  ]
}
```

### 6.2 用户模拟器模块

建议生成以下用户类型：

- 配合型用户
- 忙碌型用户
- 质疑诈骗型用户
- 明确拒绝型用户
- 信息不完整型用户
- 打断型用户
- 投诉型用户
- 诱导违规型用户
- 前后矛盾型用户
- 沉默/短回复型用户

每个模拟用户应包含：

```json
{
  "persona": "质疑诈骗型用户",
  "goal": "确认这通电话是否可信",
  "hidden_state": {
    "eligible": true,
    "main_objection": "担心诈骗",
    "will_accept_if_explained_clearly": true
  },
  "coverage_targets": [
    "真实性质疑",
    "身份核验拒绝",
    "要求说明活动来源"
  ]
}
```

### 6.3 评测指标体系

建议总分 100 分：

| 维度 | 权重 | 说明 |
|---|---:|---|
| 任务完成度 | 30 | 是否达成外呼任务目标 |
| 关键约束遵循 | 20 | 是否遵守业务规则、合规要求和禁止项 |
| 条件分支处理 | 15 | 是否正确处理拒绝、质疑、沉默、投诉等情况 |
| 多轮上下文保持 | 10 | 是否记住用户信息，是否前后一致 |
| 流程顺序与状态推进 | 10 | 是否按合理业务流程推进 |
| 话术质量 | 10 | 是否清晰、礼貌、自然、不过度施压 |
| 用户体验/满意度 | 5 | 是否降低用户困惑和负面情绪 |

### 6.4 单个评测点输出格式

```json
{
  "checkpoint_id": "identity_verification",
  "description": "客服需在办理业务前完成用户身份核验",
  "score": 0,
  "max_score": 10,
  "result": "failed",
  "evidence": "第2-4轮客服直接说明权益并尝试办理，未核验身份",
  "reason": "未满足业务办理前置条件",
  "risk_level": "high",
  "suggestion": "在开场后加入身份核验策略，并在用户拒绝提供信息时进入解释分支"
}
```

### 6.5 最终评估报告结构

建议报告包含：

1. 任务概览
2. 模拟数据概览
3. 总体评分
4. 分维度评分
5. 关键成功点
6. 关键失败点
7. 高风险违规项
8. 典型失败对话片段
9. 用户类型覆盖情况
10. 优化建议

报告示例结构：

```markdown
# 外呼数字人多轮对话评估报告

## 总体结论

本轮共生成 50 通模拟外呼对话，平均得分 76.4/100。
主要问题集中在用户质疑真实性和明确拒绝场景。

## 分维度得分

- 任务完成度：24/30
- 关键约束遵循：16/20
- 条件分支处理：8/15
- 多轮上下文保持：8/10
- 流程顺序与状态推进：7/10
- 话术质量：9/10
- 用户体验：4/5

## 主要失败原因

1. 用户质疑外呼真实性时，数字人解释不充分。
2. 用户明确拒绝后，数字人仍继续推进业务。
3. 部分对话未在结束前确认用户最终意向。

## 优化方向

1. 增加反诈/官方来源解释话术。
2. 为拒绝用户增加礼貌退出和原因记录分支。
3. 强化身份核验前置流程。
```

## 7. 本赛题推荐技术路线

推荐采用混合式评测架构：

```text
任务指令
  -> 任务拆解器
  -> Rubric / Checklist / 状态机
  -> 用户模拟器批量生成场景
  -> 数字人多轮对话
  -> 对话状态追踪
  -> 规则评测 + LLM Judge 评测
  -> 证据抽取
  -> 量化评分
  -> 可解释评估报告
```

核心原则：

- 强业务规则用规则或状态机判断。
- 语义质量、话术自然度、异议处理用 LLM Judge 判断。
- 每个评分点必须有证据。
- 每个扣分点必须能转化为优化建议。
- 用户模拟器必须覆盖异常和困难分支。
- 最终报告服务于数字人团队定位优化方向。

## 9. 最适合写进方案的一句话

本系统将复杂外呼任务指令自动拆解为可验证评测点，通过多画像用户模拟器生成覆盖性多轮对话样本，再结合业务状态机、规则校验和 LLM-as-a-judge 对完整对话轨迹进行量化评分，最终输出包含证据、扣分原因和优化建议的可解释评估报告。
