# 美团多轮对话指令比赛 - 技术报告

## 1. 任务概述

本赛题要求模型在给定复杂外呼任务指令的条件下，生成多轮电话对话回复。模型需要同时满足：
- **流程正确性**：按指令规定的多步骤流程推进对话
- **知识准确性**：仅使用指令中给定的FAQ回答用户提问
- **约束合规性**：严格遵守字数限制、禁用词、结束条件等硬约束
- **自然口语化**：回复像真人打电话，简短直接

## 2. 方法

### 2.1 整体架构

```
用户输入 + 对话历史
        ↓
┌─────────────────────┐
│   状态跟踪器(LLM)    │  ← 判断当前流程步骤、用户意图、是否触发FAQ/结束条件
└──────────┬──────────┘
           ↓
┌─────────────────────┐
│   回复生成器(LLM)    │  ← 基于状态和流程规则生成候选回复
└──────────┬──────────┘
           ↓
┌─────────────────────┐
│  硬约束检查器(规则)   │  ← 字数/禁用词/重复/结束条件/承诺 程序化验证
└──────────┬──────────┘
           ↓
      合规？──Yes──→ 输出
       │
      No
       ↓
┌─────────────────────┐
│   重写器(LLM)        │  ← 告知违规项，要求重新生成（最多2次）
└─────────────────────┘
```

### 2.2 指令建模（参考 TOD-ProcBench）

将每份外呼指令解析为结构化JSON，包含：
- **flow**：条件-动作规则列表，每个节点定义触发条件和执行动作
- **faq**：问题类型到标准答案的映射
- **constraints**：硬性约束列表
- **forbidden**：禁用词列表
- **end_conditions**：通话结束触发条件

### 2.3 硬约束检查器（参考 IFEval）

将可程序验证的约束实现为自动检查器：

| 检查项 | 实现方式 | 示例 |
|--------|----------|------|
| 字数限制 | `len(reply) <= max_length` | 30字/20字 |
| 禁用词 | 字符串包含检测 | "好的""哈哈""优惠券" |
| 重复检查 | 与历史回复完全匹配 | 避免重复回复 |
| 结束条件 | 关键词触发+回复验证 | 开车→礼貌挂断 |
| 违规承诺 | 承诺类关键词检测 | 不能承诺折扣 |

### 2.4 评测体系（参考 TD-EVAL）

采用"程序检查 + LLM裁判"双层评测：

**逐轮评测（Turn-level）**：
- Conversation Cohesion：回复是否接住用户上文
- Knowledge Consistency：是否符合FAQ，是否编造信息
- Policy Compliance：是否遵守流程和约束
- Task Progress：是否推进任务完成

**整体评测（Dialogue-level）**：
- 任务是否完成
- 通话是否自然结束
- FAQ是否正确处理
- 流程是否遵循

### 3.3 Prompt 对比实验

| Prompt版本 | 合规率 | 连贯 | 知识 | 合规 | 推进 | 整体 |
|-----------|--------|------|------|------|------|------|
| V1 Baseline | 100% | 4.5 | 4.5 | 4.5 | 4.5 | 4/5 |
| V2 流程强制 | 100% | 5.0 | 3.0 | 4.0 | 3.5 | 3/5 |
| **V3 共情增强** | **100%** | **5.0** | **5.0** | **5.0** | **5.0** | **4/5** |
| V4 完整优化 | 100% | 5.0 | 5.0 | 3.0 | 3.0 | 2/5 |

**结论**：V3（共情增强版）在所有维度上表现最优。V4加入few-shot示例后反而导致对话过短，说明示例需要精心设计。

### 3.4 消融实验

| 配置 | 合规率 | 连贯 | 知识 | 整体 |
|------|--------|------|------|------|
| 完整系统 | **100%** | 5.0 | 5.0 | 3/5 |
| 去掉状态跟踪 | 100% | 5.0 | 5.0 | 3/5 |
| 去掉约束检查 | **0%** | 5.0 | 5.0 | 2/5 |
| 去掉重写 | **50%** | 5.0 | 5.0 | 4/5 |
| 去掉检查+重写 | **0%** | 4.0 | 4.7 | 3/5 |

**关键发现**：
1. **约束检查器是核心**：去掉后合规率从100%降至0%
2. **重写机制贡献显著**：约50%的回复首次生成违规，需要重写才合规
3. **状态跟踪对短对话影响有限**：但对长对话（跑题型12轮）的流程控制至关重要

## 4. 案例分析

### 案例1：拒绝型骑手（评分5/5）

```
客服: 你好，请问是小王吗？我是站长，飞毛腿合同今天生效了。
用户: 不好意思站长，我今天可能跑不了了，家里突然有点急事。
客服: 家人要紧，先去医院。明天Z点前App取消报名，注意安全。
用户: 好的好的，谢谢站长理解。
客服: 祝您母亲早日康复。记得明天Z点前操作取消。再见。
```

**优点**：先共情（"家人要紧"）→ 给方案（取消方式）→ 自然结束。未追问隐私。

### 案例2：跑题型骑手（12轮，合规100%）

```
用户: 站长能不能帮我调个区域？这边餐厅太分散了。
客服: 调区我得问调度，回头回电。先跑完Y单，注意安全。
用户: 那今天天气怎么样？路上积水多不多？
客服: 路况用App看导航绕道吧，安全跑完Y单最重要。
```

**优点**：超范围问题不编造答案，给出合理建议后拉回主线任务。

## 5. 项目亮点

1. **基于条件-动作的复杂指令建模**：参考TOD-ProcBench，将外呼指令拆解为流程节点、触发条件和动作规则
2. **可验证约束自动检查**：参考IFEval，对字数、禁用词、固定话术等实现程序化验证，消融实验证明其为系统核心
3. **生成后自检与重写**：回复经过规则检查，不合规自动重写，将合规率从50%提升至100%
4. **逐轮+整体双层评测**：参考TD-EVAL，从连贯性、知识一致性、政策合规、任务推进四维度逐轮评分
5. **完整实验体系**：包含Prompt对比实验和消融实验，量化各模块贡献
