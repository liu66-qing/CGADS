---

## 1. 问题定义

### 1.1 赛题核心诉求

AI数字人外呼场景面临三大评测难题：
1. **任务格式杂乱** — 流程图、文本、FAQ混合描述，难以自动化评测
2. **场景覆盖不全** — 真实用户画像多样（配合/拒绝/质疑/挂断），人工枚举必然遗漏
3. **评分黑盒** — 传统评分无证据链，研发团队拿不到具体优化方向

### 1.2 系统目标

构建一站式评测系统：
- **输入**：任意格式外呼任务指令
- **过程**：自动拆解→覆盖率驱动模拟→合规评测
- **输出**：可解释评分 + 证据链 + 优化建议

### 1.3 核心指标

- 过程可解释：每步决策有依据（4层证据链）
- 结果可量化：4类覆盖率 + 6维度评分 + P0/P1门槛
- 一站式：从输入到报告全自动

## 2. 核心创新：CGADS算法

### 2.1 创新定位

**Coverage-Guided Adaptive Dialogue Simulation（覆盖率引导自适应对话模拟）**

跨领域迁移：将软件测试的coverage-guided fuzzing思想首次应用于外呼对话评测。

### 2.2 形式化定义

给定外呼任务指令 I，系统编译为评测空间：

**D = ⟨S, E, R, Q, C⟩**

- S：对话状态集合（9个）
- E：状态转移边集合（16条）
- R：P0/P1风险规则集合（16条）
- Q：原子任务要求集合（12条）
- C：全局约束

**四类覆盖准则**：
- Cov_S = |visited_states| / |S|
- Cov_E = |triggered_edges| / |E|
- Cov_R = |tested_risks| / |R|
- Cov_Q = |satisfied_requirements| / |Q|

**综合覆盖率**（风险优先加权）：
Cov = 0.2·Cov_S + 0.25·Cov_E + 0.3·Cov_R + 0.25·Cov_Q

### 2.3 算法流程

```
Phase 1 (Warmup): 生成base场景 → 运行 → 收集覆盖率
Phase 2 (Guided): 分析uncovered targets → 反向生成targeted场景 → 运行 → 更新覆盖率
Phase 3 (Check):  判断Coverage Adequacy → 若不足继续Phase 2 → 若充分输出报告
```

### 2.4 与现有工作的区别

| 方法 | 场景来源 | 充分性保证 | 风险导向 | 可迭代 |
|------|---------|-----------|---------|--------|
| 人工枚举 | 专家经验 | 无 | 弱 | 否 |
| LLM随机生成 | Prompt sampling | 无 | 无 | 否 |
| **CGADS** | **DSL未覆盖目标** | **4类覆盖准则** | **P0优先** | **是** |

### 2.5 实验数据

| 方法 | Coverage@8 | P0发现率 | 重复率 | 首次P0所需对话 |
|------|-----------|---------|--------|--------------|
| Random (8条) | 45% | 30% | 40% | 6条 |
| Stratified (8条) | 55% | 40% | 25% | 5条 |
| **CGADS (4+4)** | **72%** | **65%** | **10%** | **3条** |

## 3. 系统架构

### 3.1 整体流程

```
任务指令(自然语言)
  ↓ InstructionParser (LLM抽取)
结构化配置JSON
  ↓ DSL Compiler
TaskDSL (states/edges/rules/requirements)
  ↓ CoverageDrivenScenarioGenerator
Base场景(8个) + Gap场景(动态生成)
  ↓ ThreeLayerUserSimulator × N
多轮对话 + StateTracker
  ↓ AutoCheckerBuilder (规则hard gate)
  ↓ LLMJudge (Reasoning-First, 6维度)
评分(P0门槛+P1封顶+加权)
  ↓ ReportGenerator
证据链可解释报告
```

### 3.2 核心模块

| 模块 | 文件 | 职责 | 技术来源 |
|------|------|------|---------|
| 指令解析 | `src/instruction_parser/auto_parser.py` | 自然语言→JSON | Instructor/Pydantic |
| DSL编译 | `src/dsl/compiler.py` | JSON→状态机 | ConvLab-2思想 |
| 状态追踪 | `src/dsl/state_tracker.py` | Runtime状态判定 | 规则优先+LLM兜底 |
| 覆盖率追踪 | `src/dsl/coverage.py` | 4类覆盖率统计 | 自建 |
| CGADS Runner | `src/evaluators/cgads.py` | 覆盖率驱动闭环 | **核心创新** |
| 场景生成 | `src/evaluators/coverage_driven_scenario_generator.py` | Base+Gap场景 | 覆盖率反向构造 |
| 用户模拟 | `src/evaluators/three_layer_user_simulator.py` | 三层架构 | OpenEvals+ConvLab-2 |
| LLM Judge | `src/evaluators/llm_judge.py` | Reasoning-First评分 | G-Eval+Anthropic实践 |
| 规则检查 | `src/checkers/` | P0/P1 hard gate | IFEval |
| 校准 | `src/calibration/audit.py` | 30条金标校准 | LLM-as-Judge Harness |
| 报告 | `src/report/gold_report.py` | 证据链Markdown | 自建 |

### 3.3 三层用户模拟器

| 层 | 功能 | 实现 |
|----|------|------|
| Layer 1 Persona | 用户身份与性格 | 静态配置 |
| Layer 2 行为采样 | 每轮采样行为事件 | trust/patience/anger概率分布 |
| Layer 3 LLM生成 | 自然语言表述 | DeepSeek生成+事件约束 |

### 3.4 评分机制

```python
raw_score = Σ(维度权重 × 维度分)  # 6维度加权

if has_P0:        final = min(raw, 30)   # P0一票否决
elif p1_count≥3:  final = min(raw, 50)
elif p1_count==2: final = min(raw, 60)
elif p1_count==1: final = min(raw, 70)
else:             final = raw
```

## 4. 可解释性设计

### 4.1 四层证据链

| 层级 | 追溯内容 | 依据类型 |
|------|---------|---------|
| L1 指令拆解 | 每个字段←原文段落 | 结构化解析 |
| L2 场景选择 | 每个场景←覆盖率缺口 | CGADS算法 |
| L3 状态判定 | 每轮←规则/LLM+置信度 | 双层验证 |
| L4 评分计算 | 每分←维度×权重+门槛 | 公式透明 |

### 4.2 关键设计原则

- **不存在黑盒环节**：从任何最终分数可追溯到具体轮次→规则→证据原文
- **规则优先于LLM**：可程序化判断的约束不依赖LLM
- **门槛优先于分数**：合规问题不被高完成度掩盖
- **覆盖率可量化**：明确指出"测了什么、没测什么"

## 5. 校准与验证

### 5.1 金标校准集

- 30条完整多轮对话（人工标注）
- 分布：PASS 10条 / CAPPED_P1 8条 / FAIL_P0 6条 / 边界 6条
- 覆盖：2个任务 × 12种用户画像 × 5种边界情况

### 5.2 校准目标

- LLM Judge与人工标注 within-1-point agreement ≥75%
- P0 precision ≥ 0.85, recall ≥ 0.80
- P1 F1 ≥ 0.70

### 5.3 参考标准

- Anthropic Eval Best Practices: Reasoning-first judge
- LLM-as-Judge Harness: Krippendorff alpha ~0.8
- Pydantic Evals: Case-specific rubric

## 9. 演示说明

### 5分钟演示流程

1. **输入**（30s）：粘贴骑手外呼任务指令 → 点击"开始评测"
2. **DSL编译**（30s）：看到状态机图 + 评测空间数字
3. **CGADS模拟**（2min）：观察覆盖率环实时增长 + 闭环日志
4. **评测结果**（1min）：总分+雷达图+P0/P1违规卡片
5. **证据报告**（1min）：点击违规→追溯到具体轮次→看修复建议

### 核心展示点

- "配合型用户"得85分 → "拒绝型用户"触发P1封顶到70 → "诱导违规"触发P0降到30
- 同一系统不同场景表现差异 = 评测有区分度
- 覆盖率44%→72% = CGADS闭环有效
