## 摘要

大语言模型（LLM）的一项核心能力是遵循自然语言指令。然而，这类能力的评估尚未标准化：人工评估昂贵、缓慢，且难以客观复现；基于 LLM 的自动评估又可能存在偏差，或受限于评估模型自身能力。为克服这些问题，作者提出面向大语言模型的 Instruction-Following Eval（IFEval）。

IFEval 是一个直接、易复现的评估基准。它聚焦一组“可验证指令”，例如“写作超过 400 个词”以及“至少提到关键词 AI 三次”。作者识别了 25 类可验证指令，并构建约 500 个 prompt，每个 prompt 包含一个或多个可验证指令。论文展示了市场上两个广泛可用 LLM 的评估结果。代码和数据发布在 Google Research 仓库的 `instruction_following_eval` 目录中。

## 1 引言

大语言模型是许多前沿研究和应用的基础。LLM 的一项关键能力是遵循输入的自然语言指令，也就是通常所说的 zero-shot prompt。LLM 准确解释并遵循自然语言指令的能力非常重要：它不仅影响任务精度，也关系到实现中的安全性和可靠性。如果模型在遵循指令时出现偏差或误解，可能产生非预期输出；在医疗健康、自动驾驶等关键场景中，这类结果尤其严重。因此，确保 LLM 能够稳定遵守给定指令至关重要。评估模型表现时，评估其指令遵循能力是关键一环。

然而，评估 LLM 的指令遵循能力既复杂又困难。主要原因是人类语言天然具有主观性和歧义。同一文本可以被不同方式解释，从而导致评估模型是否遵循指令时出现不同判断。例如，当判断 LLM 回复是否遵循“用有趣的语气写作”或“生成详细推理过程但不要过度解释”等指令时，背后的标准非常不清晰。

既有评估方法大致可分为三类，各自存在缺点：

- **人工评估。** 这类方法耗时、昂贵，并依赖一组人工标注者，因此可能带来偏差和复现不一致。
- **基于模型的评估。** 这类方法使用内部或外部模型评估目标模型表现。但这种方式高度依赖评估模型自身的正确性，而这一点并无保证。如果评估模型能力有限，就会产生误导性评估信号。
- **量化基准。** 这类方法提供标准化、可扩展的评估方式。已有工作也关注生成任务，尤其是与计数相关的指令遵循。

本文提出 IFEval，用于评估语言模型遵循指令的熟练程度。该指标围绕一类称为“可验证指令”的特殊指令展开。可验证指令指的是能够客观验证是否被遵循的指令。例子包括：“写 450 到 500 个词”“整个输出应为 JSON 格式”“包含一个标题，并将其放入两个方括号中，例如 `[[ title ]]`”。

通过聚焦可验证指令，作者希望提高评估流程的清晰度和客观性，使机器模型遵循指令的能力可以被完全自动且准确地评估。此外，通过分析评估结果，研究者可以洞察哪些类型的指令通常不被遵循，并在不同指令类型上比较不同大语言模型。

需要注意的是，尽管本文聚焦可验证指令，但很少有指令能够做到 100% 客观且自动验证。总会存在一些边界情况，使判断一条指令是否被遵循变得困难。例如，对于“在邮件结尾写：P.S. I do like the cake”这一可验证指令，模型可能以 `P.S. **I do like the cake**` 结尾，其中 Markdown 的 `**` 表示加粗。此时，如果只做朴素字符串匹配，就会得到假阴性。为缓解这类问题，作者实现了相对鲁棒的验证方式，会考虑常见变体。

总体而言，作者创建了 25 类可验证指令，并进一步构建 541 个 prompt，每个 prompt 包含一个或多个可验证指令。每类可验证指令都有多个变体，包括参数变化（如写 450 到 500 个词 vs. 写 350 到 400 个词）和表述变化（如“写 450 到 500 个词” vs. “你的回复必须包含 450 到 500 个词”）。作者评估了市场上广泛使用的模型，包括 GPT-4 和 PaLM 2，并报告其基线结果。

总结来说，本文提出 IFEval：一个使用包含可验证指令的 prompt 来评估 LLM 指令遵循能力的基准。这些可验证指令是原子指令，可以用简单、可解释、确定性的程序验证对应回复是否遵循指令。作者报告了多个模型的评估结果，并公开了评估代码和 prompt。

## 表 1：25 类可验证指令

| 指令组 | 指令类型 | 简要说明 |
|---|---|---|
| 关键词 | 包含关键词 | 回复中包含关键词 `{keyword1}`、`{keyword2}`。 |
| 关键词 | 关键词频率 | 回复中单词 `{word}` 应出现 `{N}` 次。 |
| 关键词 | 禁用词 | 回复中不得包含 `{forbidden words}`。 |
| 关键词 | 字母频率 | 回复中字母 `{letter}` 应出现 `{N}` 次。 |
| 语言 | 回复语言 | 整个回复都应使用 `{language}`，不允许其他语言。 |
| 长度约束 | 段落数量 | 回复应包含 `{N}` 个段落。段落之间用 Markdown 分隔符 `* * *` 分隔。 |
| 长度约束 | 单词数量 | 用至少/大约/最多 `{N}` 个词回答。 |
| 长度约束 | 句子数量 | 用至少/大约/最多 `{N}` 个句子回答。 |
| 长度约束 | 段落数量 + 第 i 段首词 | 应有 `{N}` 个段落，段落之间只能用两个换行分隔；第 `{i}` 段必须以 `{first word}` 开头。 |
| 可检测内容 | 附言 | 在回复结尾明确添加以 `{postscript marker}` 开头的附言。 |
| 可检测内容 | 占位符数量 | 回复必须至少包含 `{N}` 个用方括号表示的占位符，例如 `[address]`。 |
| 可检测格式 | 项目符号数量 | 答案必须恰好包含 `{N}` 个项目符号，使用 Markdown 项目符号，例如 `* This is a point.` |
| 可检测格式 | 标题 | 答案必须包含一个标题，并用双尖括号包裹，例如 `<<poem of joy>>`。 |
| 可检测格式 | 选项选择 | 从给定 `{options}` 中选择一个作答。 |
| 可检测格式 | 最少高亮段落数 | 答案中至少 `{N}` 个部分要用 Markdown 高亮，例如 `*highlighted section*`。 |
| 可检测格式 | 多章节 | 回复必须有 `{N}` 个章节，并用 `{section splitter} X` 标记每节开头。 |
| 可检测格式 | JSON 格式 | 整个输出应包裹为 JSON 格式。 |
| 组合 | 重复 prompt | 首先原样重复请求，然后给出答案；重复请求前不要输出任何内容。 |
| 组合 | 两个回复 | 给出两个不同回复，两个回复之间且只能用 6 个星号 `******` 分隔。 |
| 大小写变化 | 全部大写 | 整个回复应为英文，且只使用大写字母。 |
| 大小写变化 | 全部小写 | 整个回复应为英文且全部小写，不允许大写字母。 |
| 大小写变化 | 全大写词频率 | 回复中全大写单词应至少/大约/最多出现 `{N}` 次。 |
| 开头/结尾 | 结尾检查 | 用精确短语 `{end phrase}` 结束回复，之后不得再有其他词。 |
| 开头/结尾 | 引号包裹 | 用双引号包裹整个回复。 |
| 标点 | 不使用逗号 | 整个回复中不得使用任何逗号。 |

作者选择这些指令，是因为它们要么容易验证，要么在真实应用中常见。该列表可以很容易扩展，例如加入“回复中混合两种语言”或“XML 格式”等指令。

## 2 可验证指令

可验证指令列表见表 1，包含可验证指令的 prompt 示例见表 2。本节讨论作者如何合成带有可验证指令的 prompt，以及如何计算指令遵循指标。

### 2.1 Prompt 合成

本节描述作者如何使用 few-shot prompting 和人工整理生成 prompt。所有生成的 prompt 列在附录第 6 节。

一种直接生成 prompt 的方法，是把一个基础 prompt（例如“写一篇文章”）与多个可验证指令拼接起来。但这种方法有明显问题：指令之间可能冲突。例如，一条指令把段落数限制为 5，另一条指令却要求少于 20 个词。此外，这样创建的 prompt 不够多样。很难判断被测模型是真的善于遵循某类指令，还是只擅长遵循该指令的某种特定表述。

作者通过四步流程缓解这些问题：

1. 生成一组基础 prompt，并在每个 prompt 末尾附加 1 到 3 条随机选择的可验证指令。
2. 使用 few-shot prompting 识别不合逻辑的 prompt，并将其移除。
3. 使用另一种基于 few-shot prompting 的方法改写每个 prompt，以增加表述多样性。
4. 对改写后的 prompt 逐条人工检查和编辑。

### 2.2 IFEval 指标

对于给定回复 `resp` 和可验证指令 `inst`，作者定义一个函数判断指令是否被遵循：

```text
is_followed(resp, inst) =
  True,  如果指令被遵循；
  False, 否则。
```

作者用该函数计算指令遵循准确率，并称其为 **strict metric**。

尽管可以用简单启发式和程序验证指令是否被遵循，作者仍发现会出现假阴性。例如，对“在邮件结尾写：P.S. I do like the cake”这一指令，如果模型以 `P.S. **I do like the cake**` 结尾，只做字符串匹配会误判为未遵循。为缓解假阴性问题，作者计算一种宽松的指令遵循准确率，即 **loose accuracy**。

loose 指标会对回复做若干变换，并检查任一变换后的回复是否满足指令。作者使用以下变换：

1. 移除 Markdown 中常见字体修饰符，尤其是 `*` 和 `**`。
2. 移除回复第一行，以跳过 “Sure, here it is:” 这类开场白。
3. 移除回复最后一行，以跳过 “Hope it helps.” 这类结尾语。

作者还组合任意两个变换、三个变换，以及恒等变换。因此总共有 8 种变换。

虽然 loose 验证过程减少了假阴性，但也可能引入假阳性。例如，一个不满足词数要求的回复，在删除第一行后可能被误识别为满足要求。因此，作者把 loose 标准视为原始 strict 标准的补充。

## 3 评估结果

作者评估了 GPT-4 和 PaLM 2 Small（S）。GPT-4 和 PaLM 2 S 的回复分别在 2023 年 11 月和 2023 年 8 月通过 API 调用获取。对每个模型，作者计算四个准确率：

1. **Prompt-level strict-accuracy。** 所有可验证指令都被遵循的 prompt 所占比例。
2. **Inst-level strict-accuracy。** 被遵循的可验证指令所占比例。
3. **Prompt-level loose-accuracy。** 使用 loose 标准计算的 prompt 级准确率。
4. **Inst-level loose-accuracy。** 使用 loose 标准计算的指令级准确率。

论文表 3 给出整体准确率：

| 模型 | Prompt-level strict | Inst-level strict | Prompt-level loose | Inst-level loose |
|---|---:|---:|---:|---:|
| GPT-4 | 76.89% | 83.57% | 79.30% | 85.37% |
| PaLM 2 S | 43.07% | 55.76% | 46.95% | 59.11% |

作者提醒，这两个模型参数规模差异很大，因此不能直接公平比较。

图 2 按指令类别展示了指令级 strict accuracy。总体上，GPT-4 表现显著高于 PaLM 2 S，但不同类别之间差异明显。这说明 IFEval 不仅能给出总分，也能帮助研究者定位模型在哪些指令类型上容易失败。

## 4 讨论与未来工作

本文提出使用一组可验证指令来评估 LLM 的指令遵循能力。IFEval 是一种易复现、无偏、自动化的方法。

尽管有上述优势，当前 IFEval 实现仍可在多个方面改进，尤其包括：

1. 增加可验证指令的多样性和数量。
2. 扩展到多模态用例。例如，“生成至少 3 张……的图像”。

作为未来工作，作者计划改进 prompt 和可验证指令，使它们更贴近真实应用。此外，作者计划用更多可验证指令扩展该方法，并支持多模态用例。
