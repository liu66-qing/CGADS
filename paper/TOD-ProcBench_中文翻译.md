## 摘要

在真实世界的任务型对话（task-oriented dialogue, TOD）场景中，例如旅游预订、银行业务和医疗健康的客服支持，智能体需要在与客户进行多轮对话时严格遵守复杂指令。这些指令通常以自然语言形式呈现，既包含一般性准则，也包含带有复杂约束的分步骤流程。现有 TOD 基准往往把复杂指令简化为由意图、槽位和 API 调用配置组成的简单模式，从而过度简化了真实指令的复杂性。

为弥补这一空缺，并系统评测大语言模型（LLM）的指令遵循能力，我们提出 TOD-ProcBench。它是一个具有挑战性的基准，包含带有细粒度复杂约束的流程指令，用于评估不同 LLM 在多轮任务型对话中理解并遵循指令的能力。该基准数据集由高质量 ABCD 数据集衍生出的指令文档以及经过人工质量控制的对应对话组成。我们把细粒度约束和动作流程表述为多层级的“条件-动作”指令语句。

我们设计了三项任务，用于全面评测 LLM 在多轮 TOD 中遵循复杂指令的能力。任务 1 评估 LLM 如何从复杂指令中检索最相关语句，并预测对应的下一步动作。任务 2 通过注入不一致性并操纵原始指令来合成违反指令的回复，然后分析 LLM 识别违规回复的效果。任务 3 考察 LLM 在给定原始复杂指令时，条件式生成合规回复的能力。

基准结果显示，在所有三项任务中，LLM 在多轮对话里的指令遵循能力仍存在显著差距。此外，我们还研究了多语言设置和不同指令文本格式对合规表现的影响。作者根据 Llama 3.3 Community License Agreement 发布该基准，以推动多轮 TOD 复杂指令遵循能力研究。数据集可从 Amazon Science 页面下载。

## 1 引言

任务型对话系统在与用户交互时需要采取明确步骤和动作，其主要目标是满足客户请求。在多轮交互中，系统首先理解用户查询，然后在遵守复杂指令所定义约束的同时，收集解决请求所需的信息。复杂指令包含严格准则，规定动作和 API 调用必须在什么条件下执行。指令复杂度会随用户意图和任务领域而变化。

现有数据集要么没有充分体现多轮对话带来的挑战，要么没有捕捉真实世界指令的复杂性，缺少图 1 所示的细粒度条件式指令。真实世界的指令通常使用自然语言的 IF（条件）- THEN（动作）结构。与符号化格式相比，这类表达会引入一定歧义，但也能表达更复杂的指导规则。因此，研究社区需要公开数据集，包含遵循细粒度自然语言条件指令的多轮对话。

图 1 展示了一个额外费用场景：ABCD 数据集中只给出工作流动作顺序，例如拉取账户、验证身份、查询系统、询问会员等级、更新订单等；但对应的真实指令还会区分是否为公司错误、用户会员等级是金/银还是铜/访客、是否可以移除费用、是否可升级给经理等细节。ComplianceScorer 会错误地把多个回复都判为合规，尽管其中一些实际上违反了费用移除规则。这说明，基本工作流不足以表达细粒度条件约束。

自然语言指令还有一个额外好处：同一套一致的指令可以无缝共享给人工客服和聊天机器人，而不需要转换为另一种格式。基于这一目标，作者构建了一个名为 TOD-ProcBench 的多语言数据集，包含对话以及条件-动作格式的复杂指令。本文贡献如下：

- 提出 TOD-ProcBench。它基于 ABCD 数据集构建，包含复杂的条件-动作自然语言指令，以及多语言多轮对话。
- 发布三类 TOD 任务基准：一是“指令与下一动作选择”，衡量 LLM 在多轮对话中识别适用指令的能力；二是“指令遵循评估”，衡量 LLM 检测复杂指令违规的能力；三是“指令遵循回复生成”，衡量 LLM 在面向完整对话交互的复杂指令下生成回复的能力。
- 提出一种生成违反指令回复测试基准的方法：对指令注入不同扰动，并提示 LLM 生成与真实回复相似但遵循被操纵指令的回复。
- 实验表明，本文研究的所有 LLM 在三类任务上都仍然困难重重：复杂格式指令选择、违规回复评估，以及遵循指令的回复生成。此外，LLM 在这些任务上展现出相近的多语言能力，但对英语略有偏好。

## 2 相关工作

一般而言，客户服务聊天机器人、实体机器人和虚拟助手等任务型对话系统，在执行 API 调用和动作以解决用户问题时，都必须严格遵守预定义流程。已有数据集通过 schema-guided 范式为 TOD 引入约束。合成数据集 SGD 的 schema 通过必要槽位集合及其填充顺序来约束动作执行。附录 A 显示，SGD 中“查找下一班航班”的意图 schema 包含出发地、目的地、机场、座位等级等槽位，以及每个意图所需的槽位。

类似地，STAR 和 FLAP 以流程图格式呈现 TOD 中的 schema 概念。对这些数据集来说，包含动作和槽位值的 schema 通常不能反映真实世界指令的性质，因为真实指令往往具有更复杂的分步骤结构和条件-动作组合。

本文所基于的 ABCD 数据集包含人工客服需要按预定义顺序执行的准则。本文进一步提出更真实的自然语言指令，以多层级条件-动作语句的形式，精确说明执行动作所需满足的条件。近期也有数据集和研究评测 LLM 智能体遵循 SOP 的能力，但它们更关注工具选择准确率；本文则补充研究多轮对话设置中的会话能力。

## 3 TOD-ProcBench

### 3.1 问题概述

在本文设定中，给定某种语言 `l` 中一段对话的历史 `u`，例如“user: i need to return this jacket. it is the wrong size”，任务型对话系统 `D` 要生成回复 `r`，必须遵循给定指令 `n` 中的一组条件 `c ⊆ C`，并执行相应动作序列 `a = <a1, a2, ..., an>`，其中每个动作 `ai ∈ A`。

TOD-ProcBench 有两个主要维度：一是指令格式维度 `f`，包含三种英语自然语言格式：`f1 = Nested If-Then`、`f2 = Flattened If-Then`、`f3 = Flattened JSON Mappings of Conditions and Actions`；二是对话语言维度 `l`，包含阿拉伯语、中文、英语、法语、德语、印地语和西班牙语。

### 3.2 复杂指令组件

#### 3.2.1 条件

条件是执行对应动作序列之前必须验证的具体状态或属性，例如客户属性（会员等级）、商品属性（购买日期）或对话状态（用户意图）。

#### 3.2.2 动作

当相关条件满足时，TOD 必须执行具体操作或回复。动作与条件按层级组织。例如，当用户询问退货政策（一层条件）时，智能体必须先识别用户会员等级（一层动作）。如果用户是银卡会员（二层条件），智能体再提供具体信息，例如“只有过去 6 个月内购买的商品可以退货”（二层动作）。

#### 3.2.3 组合

与既有工作类似，作者使用六种组合技术从基础条件和动作组件中构造完整复杂指令：Single、And、Or、Chain、Selection 和 Nesting。

### 3.3 指令文档构建

TOD-ProcBench 从 ABCD 数据集衍生而来。ABCD 是一个人工撰写对话集合，由标注者按照预定义工作流动作创建。作者聚合 ABCD 中具有相同用户意图的所有人-人对话，这些对话天然包含相似的对话轨迹。附录 B 给出了三种复杂指令格式 `f1`、`f2`、`f3` 的示例和符号表示。

**格式 f1。** 作者使用 LLM 和 1-shot 示例，为 ABCD 中 55 个用户意图分别生成 “Nested If-Then” 复杂指令文档。每个意图平均对应 146 段对话。作者用平均条件分支因子 2.2、最多 4 层嵌套条件等特征来量化 TOD-ProcBench 的复杂性。为回答“复杂自然语言指令的格式如何影响 LLM 合规能力”，作者以 `f1` 为基础派生出另外两种指令格式；这些格式仍以英语自然语言文本呈现基础条件和动作。

**格式 f2。** “Flattened If-Then” 将 `f1` 中嵌套的 If-Then 结构展平为一系列单层条件-动作 If-Then 语句。作者用该格式评估 LLM 是否对 `f1` 中存在而 `f2` 中不存在的 Nesting 和 Chain 组合技术有强偏好。`f2` 的展平结构把 Chain 降解为一系列带 And 的复合条件。

**格式 f3。** “Flattened JSON Mappings of Conditions and Actions” 将 `f2` 中自然语言 If-Then 语句转换为结构化 JSON 对象，其中包含若干映射列表，每个映射表示需要考虑的相关条件和动作。

### 3.4 扩展到多语言对话

作为 TOD-ProcBench 构建工作的实际应用，并作为评估 LLM 复杂指令遵循多语言能力的第一步，作者希望回答：“英语指令是否仍能有效指导另一种语言中的多轮对话？” 这一研究有助于判断，在全球市场中使用英语指令的生产环境里，是否有必要为其他语言构建高质量指令。

作者只把原始 ABCD 数据集中的多轮对话部分翻译为另外六种语言：阿拉伯语、中文、法语、德语、印地语和西班牙语。指令本身仍保持英语。

### 3.5 质量验证

作者请人工标注者评估 TOD-ProcBench 中英语对话与 55 个 `f1` 格式生成文档之间的配对质量。标注者在附录 A 图 3 所示界面中回答两个是/否问题：

**准确性。** 生成指令是否准确反映了对话中的信息？如果答案是否，标注者需要指出指令中哪部分不正确。

**缺失。** 对话中是否有重要信息没有出现在模型生成的指令中？如果答案是，标注者需要指出对话中哪部分没有进入指令。

每个对话-指令配对由三名标注者评估，总计 3,012 条标注。63% 的对话获得完全一致标注；多数投票用于确定“准确性”（82%）和“缺失”（15%）评估。准确性排名前三的意图是 recover_password、recover_username 和 search_results，它们共享相似对话轨迹；排名后三的是 shirt、timing 和 policy，这些意图可能包含范围更广的查询。作者仅关注 769 个高质量配对。

以英语对话和 `f1` 英语指令配对作为 TOD-ProcBench 基础版本后，作者沿指令格式和对话语言两个维度扩展数据集。对于格式扩展和多语言扩展数据集，作者使用 LLM judge 评估质量；格式变换正确率为 100%，各语言变体正确率为 98.3% 到 99.8%。

## 4 任务

在三项任务、三种指令格式和七种对话语言的所有实验中，作者提示六个 LLM 先生成逐步推理，再生成相应输出，即使用 chain-of-thought（CoT）：Qwen3-14B、Llama3.3-70B、Gemma3-27B-IT、Claude3.5-Sonnet-V1、Claude3.5-Sonnet-V2 和 Claude3.7-Sonnet。未在正文表格中呈现的实验结果放在附录 D。

### 4.1 任务 1：指令检索与下一动作预测

任务 1 评估 TOD 根据对话历史检索相关指令并预测合适下一动作的能力。对于 ABCD 数据集中的全部 30 个动作，作者定义触发条件，用于决定每个动作应在何时执行。作者还引入一个空动作，用于处理客户目标已经达成、智能体不需要进一步动作的场景。

#### 4.1.1 数据集

为评估模型在用户-智能体交互不同阶段的表现，作者从给定对话中抽取所有以客户轮次结束的部分对话转写。例如，一段有四个客户轮次的对话，会产生四段部分对话，每段都以客户轮次结尾。然后，作者把每段部分对话与其对应指令配对。总体而言，作者从 1,004 段测试对话中抽取了 6,953 段部分对话。

为了检查模型对智能体下一动作预测的准确性，作者处理每段部分对话历史对应的 ABCD 工作流步骤列表，识别与现有智能体轮次最相关的动作。剩余工作流步骤中的第一个动作被用作真实下一智能体动作。任务 1 输入示例见附录 A。

#### 4.1.2 结果

作者提示六个 LLM 在给定对话历史、完整指令列表、可能智能体动作以及动作调用条件时，生成下一智能体轮次，并给出 top-k 最相关指令，其中 `k = 1` 或 `k = 2`。计算 top-2 准确率时，只要 LLM 检索出的两个相关指令中任意一个是真实指令，即视为正确。所有模型均采用 few-shot in-context learning 范式，并提供三个示例。

表 1 给出了模型预测正确指令和智能体下一动作的准确率。较大的 LLM 在 `k = 1` 和 `k = 2` 下表现相对更稳定；所有 LLM 在 top-2 场景下，无论整体准确率还是仅指令检索准确率都更高。多数 LLM 偏好格式 `f1`。所有 LLM 在正确预测智能体下一动作方面都更困难，这可能源于动作之间的细微区别以及触发条件的重叠。

作者还根据给定智能体回复在对话中的相对位置计算模型准确率：把轮次按所在位置划为开头、中段或结尾三分之一。表 1 显示，较大模型在整段对话中表现较稳定。虽然较小模型随着从对话历史中获得更多信息，在指令检索和动作预测上会变好，但所有 LLM 仍有明显提升空间。附录 D 显示，在各种对话语言中，较大模型通常优于较小模型，但所有 LLM 的表现并未受到语言严重影响。

**表 1 概要。** 在英语对话中，Claude3.7-Sonnet 整体表现最佳，top-1/top-2 overall 为 0.3948/0.4310，指令检索为 0.8267/0.8943。相比之下，下一动作预测显著更难。模型的最佳指令格式并不完全一致，但 `f1` 与 `f2` 更常表现较好。

**表 2 概要。** 在多语言设置中，Claude3.7-Sonnet 仍通常取得最高准确率。英语、法语、德语、西班牙语、中文、阿拉伯语和印地语之间差异存在但并不剧烈，说明对话语言变化没有摧毁模型完成该任务的能力。

### 4.2 任务 2：合规性评估

任务 2 的主要目标是评测 LLM 对给定 TOD 回复相对于对话历史和智能体必须遵循的指令进行合规性打分的能力。这种能力既可作为推理期间的质量验证器，也可作为改进指令遵循型 TOD 开发流程的代理指标。作者将其形式化为二分类任务：区分遵循指令的智能体回复和违反指令的智能体回复。

**数据集。** 在合规性评估任务中，作者只关注与指令直接相关的智能体话语。为了从问候、确认客户信息、纯共情等通用回复中分离这些回复，作者使用 LLM 对任务 1 数据集 6,953 段部分对话中的智能体轮次进行相关/不相关分类。

**合规性操纵流程。** 为构建二元“遵循指令/违反指令”数据集，作者使用人工撰写的 ABCD 回复作为遵循指令示例，并通过四步流程生成违反指令的对应样本：一是抽取相关指令片段；二是操纵抽取出的指令；三是基于被操纵指令生成违反指令的回复；四是验证生成的违规回复是否蕴含原始真实回复。

**1. 指令片段抽取。** 作者把对话历史、指令文档和真实遵循指令的智能体回复作为输入，提示 LLM 选择一段与真实回复直接相关的指令片段，并包含其最近的 if 条件。如果找不到这样的相关指令部分，则排除该回复样本。

**2. 指令操纵。** 作者使用 LLM 对抽取出的指令片段和完整源指令进行系统性操纵，以模拟以下违规类别：

- 细粒度参数不匹配：把指令片段中的数值（例如客户可退货天数）替换为随机生成数字，模拟细微合规违规。
- 单条件动作替换：对包含一个条件和多个动作的指令，随机把原动作替换为其否定或同一指令中的另一个替代动作，捕捉动作层面的指令违规。
- 多条件动作替换：对包含多个条件组合的指令，改变逻辑条件-动作关系，例如把普通会员退货政策用于金卡会员，从而违反动作对对应条件的真实依赖关系。

**3. 回复生成。** 作者向 LLM 提供对话历史、被操纵指令、原始指令和真实回复，并提示它最小化修改真实回复，使其覆盖原始指令和被操纵指令之间的主要差异。提示中说明生成回复必须遵循被操纵指令。这隐含保证了生成回复相对于原始指令而言是违规回复。

作者使用 LLM 验证生成的违规回复不会蕴含真实回复。在 2,288 个生成样本中，307 个需要额外操纵循环。人工验证确认 87% 的指令违规是准确的，共 1,982 个样本，最终得到 3,964 个平衡样本（合规与不合规）用于任务 2。

**结果。** 作者把六个 LLM 作为 TOD-ProcBench 中指令违规的二分类器进行比较，采用两种方法：

- 方法 1：给定对话、下一智能体回复和相关指令片段，LLM 将智能体下一回复分类为合规或违反指令。
- 方法 2：LLM 首先把给定智能体回复归因到最相关指令，然后验证识别出的指令与原始提供的真实指令是否相互蕴含。蕴含表示回复合规。

任务 2 使用准确率作为评估指标。表 3 显示，多数模型在两种方法下对英语对话表现略好。表 3 也展示了方法 2 对合规性评估表现的影响：所有模型在依赖指令层比较时表现都更差。方法 2 会先要求模型基于给定对话生成相关指令，然后再用生成指令验证合规性，因此提高了任务复杂度。附录表 32 同样表明，LLM 尚不能准确完成合规性评估任务，并且对格式 `f1` 有轻微偏好，这需要进一步研究。

**表 3 概要。** 在 `f1` 指令格式下，Claude3.7-Sonnet 在各语言和两种方法中通常最好，英语为 0.7608/0.7298。方法 1 普遍优于方法 2。较小模型或开源模型在不同语言中下降更明显，尤其 Llama3.3-70B 在方法 2 中表现较弱。

### 4.3 任务 3：合规回复生成

受真实世界条件生成场景启发，任务 3 考察 TOD 基于预定义指令进行条件生成的能力，这些指令要求回复遵循指定准则。

**数据集。** 在任务 3 中，作者只考察 TOD 在此前对话已经遵循给定指令时生成后续回复的能力。作者使用任务 2 流程第一步中的 3,310 段部分对话，它们包含遵循指令的回复、对话历史和对应指令。

**结果。** 作者要求六个 LLM 在遵循给定指令的同时继续对话。在提示中，作者把问题分解为：识别智能体已经遵循的步骤；基于给定指令选择下一最相关动作；生成匹配检索动作的回复。作者使用 Claude3.7-Sonnet（任务 2 表现最好的 LLM）作为 judge，在不知道真实答案的情况下评估回复质量和合规性。根据其合规分类，作者计算任务 3 的合规率，分数越高表示越能持续遵守指令。

表 4 比较了模型生成质量。较大的 LLM 在不同对话语言和指令格式维度上通常比小模型合规性更高。一些较小模型往往不按指定回复格式回答，导致大量不合规回复和很低合规率。所有 LLM 的表现没有受到语言显著影响。作者观察到，模型通常会生成更长话语，因为它们很难从指令中准确识别一个下一动作并只覆盖该动作。尽管遵循指令的生成在语义上未必接近真实回复，Claude3.7-Sonnet 仍会把它们判为合规。

**表 4 概要。** Claude3.7-Sonnet 和 Claude3.5-Sonnet-V2 在所有语言中合规率约为 0.95 左右，明显高于 Qwen3-14B、Llama3.3-70B 和 Gemma3-27B-IT。Gemma3-27B-IT 在英语上较好，但在其他语言中降至约 0.30；Qwen3-14B 整体约 0.22-0.24。

## 5 结论

本文把复杂条件-动作格式的自然语言指令应用于任务型对话，以更接近真实世界复杂指令的复杂性。基于这些指令文档及其在多语言对话中的应用，作者提出三项相关任务：指令检索与下一动作预测、指令遵循评估、合规回复生成。作者在 TOD-ProcBench 中发布每项任务的基准，并显示先进 LLM 在所有这些任务上仍难以取得理想表现；同时，在更具挑战性的细粒度任务中，多语言能力仍需增强。

作为未来工作，作者计划把 TOD-ProcBench 扩展到更多数据集格式，包括跨领域特定指令的代码化结构。
